# AGENTS.md

## 1. 项目目标与范围

- 本项目是一个在 macOS / Linux 上开发的视觉小说 demo。
- 风格参考：赛博酒馆、霓虹夜景、对话驱动、分支叙事。
- 开发方式是边做边学，优先逐步交付，不追求一次性生成完整游戏。
- 当前优先级：项目骨架、最小可运行 demo、可解释、可扩展。

## 2. 当前技术栈与依赖

- 已确认系统工具：`Homebrew`、`git`、`python3`、`node`、`ffmpeg`、`ImageMagick`、`ripgrep`、`jq`。
- Python 虚拟环境：根目录已有 `.venv`。
- 当前已知最小 Python 包：`pillow`、`pyyaml`。
- 目前仓库里尚未确认具体游戏引擎；在未明确前，不假设 Ren'Py、Godot 或 Web 框架已经接入。

## 3. 目录结构约定

- `docs/`：设计说明、开发计划、学习笔记。
- `tools/`：本地开发脚本、资源处理脚本、检查工具。
- `assets/`：图片、音频、视频、字体等原始或导出资源。
- `data/`：剧情脚本、分支结构、配置、结构化数据。
- `game/`：最小可运行 demo 的运行时代码与游戏内容。
- 根目录：仅保留项目入口文档、环境检查脚本和仓库级约束文件。

## 4. Codex 的工作方式约束

- 先读现有文件，再改动；不要凭空假设仓库已有功能。
- 单次任务只推进一个明确目标，避免同时扩展玩法、剧情、美术流程。
- 优先做小步可运行改动，不直接铺开完整内容生产。
- 不删除用户已有文件，不重写无关结构，不修改 shell 配置文件。
- 信息不足时做保守假设，并在输出中明确写出假设。

## 5. 每次任务的输出格式

- 先说明将读取哪些文件、将修改哪些文件。
- 实施后给出：改了什么、为什么这么改、如何验证。
- 若未执行某一步，直接说明原因与阻塞点。
- 输出保持简短，优先给可执行命令和文件路径。

## 6. 修改代码前必须做的检查

- 查看根目录和相关子目录当前结构。
- 阅读目标文件现有内容；若文件不存在，再新建。
- 检查是否已有相同脚本、配置或文档，避免重复创建。
- 如果任务涉及依赖或运行方式，先核对 `README.md`、`setup_check.sh` 和 `.venv` 状态。

## 7. 修改代码后必须做的验证

- 对文档或脚本改动，至少执行一次相关检查命令。
- 对环境相关任务，运行 `./setup_check.sh`。
- 对代码任务，至少验证入口能运行或静态检查能通过。
- 如果无法验证，必须明确写出未验证项和原因。

## 8. 面向“边做边学”的说明要求

- 每次只解释当前任务直接相关的概念，不展开成教程大全。
- 说明要覆盖：目标、关键文件、运行路径、下一步建议。
- 优先解释“为什么现在这样做”，再解释“以后如何扩展”。
- 新增脚本或配置时，尽量让命名直观、注释简短。

## 9. 视觉小说 demo 的最小范围定义

- 一个可启动的最小场景。
- 一段对话流程，包含角色发言与基础文本展示。
- 至少一个分支选择，并能导向两个不同结果之一。
- 使用占位资源是允许的，但结构必须可替换为正式资源。
- 数据与展示逻辑尽量分离，便于后续扩展剧情和工具链。

## 10. 暂时不要做的事情

- 不一次性生成长篇剧本、完整世界观或大量美术资源。
- 不引入未经确认的重型引擎、构建系统或数据库。
- 不提前做复杂 UI 动效、存档系统、成就系统、多语言系统。
- 不为了“未来可能需要”而过度抽象。
- 不把学习型 demo 直接推向生产级架构。
