# 最小可运行视觉小说 demo 实施计划

保守假设：当前仓库尚未确定具体游戏引擎，因此先按“本地浏览器可运行的 HTML/CSS/JS demo”规划最小闭环。目标是完成 1 个主场景、2 个角色、1 个分支选择、1 个状态变量、1 套基础 UI 风格，不依赖完整美术资源。

## 第 1 步：确定最小数据结构与运行入口

- 要改哪些文件：
  - `docs/demo_plan.md`
  - `data/scene_intro.json` 或 `data/scene_intro.yaml`
  - `game/index.html`
  - `game/main.js`
- 为什么先做这一步：
  - 先把“内容长什么样”和“程序从哪启动”定下来，后续 UI 和分支逻辑才不会反复推翻。
- 做完后我能学到什么：
  - 视觉小说最小 demo 的核心不是美术，而是场景数据、对话节点和入口脚本之间的关系。
- 如何验证是否成功：
  - 打开 `game/index.html` 后，页面能加载一个最小占位界面。
  - `main.js` 能读到一份场景数据，并在控制台打印或渲染第一句对白。

## 第 2 步：搭建基础 UI 骨架与视觉风格

- 要改哪些文件：
  - `game/index.html`
  - `game/styles.css`
  - 可选：`assets/` 下的占位背景或配色说明
- 为什么先做这一步：
  - 在最小数据入口确定后，先做 UI 骨架，能尽快看见“赛博酒馆、霓虹夜景”的呈现方向。
- 做完后我能学到什么：
  - 一个视觉小说最基础的界面通常只需要背景层、角色名、对白框、选择区，不需要先做复杂动画。
- 如何验证是否成功：
  - 页面能显示稳定布局：背景、标题或场景名、对白框、按钮区。
  - 即使只用纯色、渐变或占位图，也能看出基础风格。

## 第 3 步：接入 2 个角色和顺序对话播放

- 要改哪些文件：
  - `data/scene_intro.json` 或 `data/scene_intro.yaml`
  - `game/main.js`
  - 可选：`game/styles.css`
- 为什么先做这一步：
  - 先跑通线性对话，能把“角色切换、文本显示、下一句推进”这条主链闭环起来。
- 做完后我能学到什么：
  - 角色信息、对白内容、推进逻辑如何分离，方便以后替换文本或扩写剧情。
- 如何验证是否成功：
  - 页面能依次显示 2 个角色的对白。
  - 点击“下一句”或按键后，文本会稳定推进到下一条。

## 第 4 步：加入 1 个状态变量

- 要改哪些文件：
  - `data/scene_intro.json` 或 `data/scene_intro.yaml`
  - `game/main.js`
  - 可选：`docs/` 中的状态说明文档
- 为什么先做这一步：
  - 分支叙事依赖状态；先加一个最小状态变量，比直接写复杂条件分支更稳。
- 做完后我能学到什么：
  - 状态变量如何影响后续对白或选择结果，比如“信任值”“是否点了某杯酒”等。
- 如何验证是否成功：
  - 在控制台或页面调试区能看到状态变量初始值与变化结果。
  - 状态更新不会打断现有对白流程。

## 第 5 步：加入 1 个分支选择

- 要改哪些文件：
  - `data/scene_intro.json` 或 `data/scene_intro.yaml`
  - `game/main.js`
  - `game/index.html`
- 为什么先做这一步：
  - 线性对话和状态已经通了，再加一个选择，最容易形成“视觉小说 demo”而不是“文本播放器”。
- 做完后我能学到什么：
  - 选择项如何更新状态，并跳到不同节点或不同结尾。
- 如何验证是否成功：
  - 页面出现 2 个可点击选项。
  - 选择不同选项后，会看到不同的后续对白或不同结局文案。

## 第 6 步：整理最小资源占位与可替换结构

- 要改哪些文件：
  - `assets/` 下的占位资源
  - `game/styles.css`
  - `README.md`
- 为什么先做这一步：
  - 在逻辑跑通后再整理资源占位，可以避免为尚未稳定的结构做无效美术准备。
- 做完后我能学到什么：
  - 如何把背景、角色名牌、颜色变量等做成可替换结构，而不是把资源路径写死在各处。
- 如何验证是否成功：
  - 替换一张背景图或修改一组颜色变量，不需要重写主逻辑。
  - `README.md` 能说明资源应放在哪里、如何替换。

## 第 7 步：补齐运行说明与最小验证流程

- 要改哪些文件：
  - `README.md`
  - `tools/` 下可选的本地启动脚本
  - `docs/demo_plan.md`
- 为什么先做这一步：
  - 最小 demo 做完后，需要把“怎么运行、怎么验证、下一步扩展什么”固定下来，方便继续边做边学。
- 做完后我能学到什么：
  - 一个小型 demo 也需要清晰的运行入口、验证步骤和后续迭代边界。
- 如何验证是否成功：
  - 新人进入仓库后，按文档步骤能启动 demo。
  - 文档能清楚说明当前只完成了哪些最小能力。
